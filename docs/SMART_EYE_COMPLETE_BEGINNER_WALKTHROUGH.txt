SMART EYE COMPLETE BEGINNER WALKTHROUGH
=======================================

Document purpose
----------------

This text file is written for a colleague who wants to understand the Smart Eye
project from the beginning, even if they are not a professional programmer or AI
engineer yet.

The goal is to explain the full story:

1. Why the project exists.
2. How the AI model part started from a Kaggle dataset.
3. How the object detection model was trained and exported.
4. How face recognition works.
5. How the desktop app was built.
6. How the database stores everything.
7. How the backend processes cameras and AI results.
8. How the frontend shows everything to the user.
9. How alerts, logs, analytics, clips, reports, accounts, and settings work.
10. Which files are responsible for which part of the system.

Important honesty note:
The repository contains the finished application code and the exported ONNX
object detection model. It does not contain the original Kaggle dataset files or
training notebooks. That is normal because datasets and training runs are often
large and are not committed to the final app repository. The object model file
itself contains metadata saying it is an Ultralytics YOLO11s detection model
trained on "yolo_safety_dataset/data.yaml". The project story says the dataset
came from Kaggle. So this guide explains that training workflow clearly, then
explains the actual app code exactly as it exists in this repository.


0. Beginner learning path for this project
-----------------------------------------

This project has two main AI systems:
- object detection (OD)
- face recognition (FR)

It also has a plugin design so object detectors can be registered and reloaded.
The goal of this section is to give you a complete beginner path from zero to
understanding this exact app.

A. Start with the basics
------------------------

1. Learn Python fundamentals:
   - variables, functions, loops, conditionals, and data structures.
   - the app is written in Python, so understanding Python is the first step.

2. Learn NumPy basics:
   - NumPy arrays are used to represent images and face embeddings.
   - images become arrays of shape [height, width, channels].
   - embeddings become arrays of shape [512] or similar.

3. Learn basic computer vision ideas:
   - images are pixels and colors.
   - video is many frames per second.
   - detection means finding objects and drawing boxes around them.

B. Learn the object detection model flow
----------------------------------------

1. Understand object detection:
   - object detection finds objects in an image.
   - it returns bounding boxes, class labels, and confidence scores.
   - typical output looks like: [x1, y1, x2, y2, class, score].

2. Learn YOLO basics:
   - YOLO means "You Only Look Once." It predicts all objects in one pass.
   - YOLO divides an image into a grid and predicts boxes and classes for each
     cell.
   - the advantage is speed: it can run in real time on video.

3. In this project, YOLO is the training/original architecture:
   - the shipped object model is a YOLO-style detector.
   - it was trained/fine-tuned on a safety dataset.
   - the model file is exported to ONNX and used by the app.

4. Learn ONNX:
   - ONNX stands for Open Neural Network Exchange.
   - it is a common file format for trained models.
   - ONNX lets the app run the model without the original training framework.
   - the app uses ONNX Runtime to load and execute the model.

5. Understand the difference: training vs inference
   - training is how the model learns from labeled images.
   - inference is how the model makes predictions on new images.
   - Smart Eye only does inference at runtime.
   - the ONNX model is already trained before the app runs.

C. Learn face recognition basics
--------------------------------

1. Understand face detection vs recognition:
   - face detection finds faces in an image.
   - face recognition identifies whose face it is.

2. Learn embeddings:
   - an embedding is a numerical vector representing a face.
   - similar faces produce similar embeddings.
   - comparing embeddings is how recognition works.

3. This project uses InsightFace for the face system:
   - InsightFace is a pretrained face recognition package.
   - it detects faces and creates embeddings.
   - Smart Eye stores known people as embeddings in SQLite.
   - when a new face appears, the app compares its embedding to stored ones.

4. Learn similarity and thresholds:
   - embeddings are compared using similarity scores.
   - if the score is above a threshold, the face is recognized.
   - if not, the face is treated as unknown.

D. Learn how this app combines both systems
-------------------------------------------

1. Camera frames arrive from webcams, RTSP, or video files.
2. A detector manager processes each frame.
3. The object model finds safety objects and violations.
4. The face model detects faces and identifies known people.
5. The results are converted into state objects for rules and alarms.
6. The UI displays live cameras, detections, recognized people, and alerts.

E. Learn the plugin and reload behavior
---------------------------------------

1. The app uses a plugin system for OD models:
   - `model_plugins` stores registered detection models.
   - `plugin_classes` stores the classes each model can detect.
   - `camera_plugins` stores which cameras use which models.

2. The default shipped model is `data/models/Obj-Detection.onnx`.
3. Reload Plugins is used when model settings change:
   - it refreshes which models are enabled.
   - it reloads the ONNX plugin state without restarting the app.
   - it is useful after changing plugin assignments or fixing a model.

F. Learn the exact project files to read next
--------------------------------------------

Start with these files after the concepts are clear:
- `backend/models/onnx_object_model.py`
  - loads ONNX object detection models
  - selects GPU or CPU provider
  - warms up the model

- `backend/models/face_model.py`
  - loads InsightFace face recognition models
  - detects faces and computes embeddings
  - identifies known faces

- `backend/models/model_loader.py`
  - manages the global face model and plugin models
  - provides a single loader interface for the app

- `backend/pipeline/detector_manager.py`
  - processes frames from cameras
  - runs object detection and face recognition in parallel
  - builds detection state for rules and alarms

- `utils/embedding_utils.py`
  - converts embeddings to bytes for database storage
  - converts bytes back to embeddings for comparison

- `backend/database/schema.sql`
  - shows how model plugins, plugin classes, and camera assignments are stored
  - shows how known faces and embeddings are stored

G. What to remember about this project
--------------------------------------

- Smart Eye is primarily an inference app, not a training app.
- The object detection model is a trained YOLO-like model exported to ONNX.
- The face recognition model is a pretrained InsightFace model.
- The app loads models, processes frames, and evaluates rules.
- The ONNX model is used because it is portable and can run on different hardware.
- The face model is used because embeddings are the best way to recognize people.

H. If you want the simplest explanation
--------------------------------------

- The app watches cameras.
- It uses a trained object detector to find safety objects.
- It uses a trained face model to recognize people.
- It stores known people as numeric embeddings.
- It uses ONNX so the object model can run without the training framework.
- It uses InsightFace so face recognition works with pretrained face embeddings.

1. The project in one simple paragraph
--------------------------------------

Smart Eye is an intelligent safety surveillance desktop application. It lets a
user add cameras or videos, run AI detection on frames, recognize known faces,
check safety equipment such as hardhats, masks, and safety vests, create rules,
trigger alarms, save logs and snapshots, review recorded clips, and generate
analytics reports. It is built as a Python desktop app using PySide6 for the
interface, OpenCV for video handling, ONNX Runtime for object detection, InsightFace
for face recognition, and SQLite for local storage.


2. What problem Smart Eye solves
--------------------------------

The problem is simple:

In places like factories, labs, warehouses, construction areas, schools, or
restricted offices, people may need to follow safety or access rules. For example:

- A person must wear a hardhat.
- A person must wear a mask.
- A person must wear a safety vest.
- Unknown people should be flagged.
- Some known people should be allowed only in specific cameras or areas.
- Managers need logs and reports to review what happened later.

Doing this manually is tiring and error-prone. A human guard cannot watch every
camera every second. Smart Eye helps by watching camera frames, running AI models,
and turning the results into useful events: alarms, logs, clips, snapshots, and
analytics.

4. Object Detection in this project
-----------------------------------

What object detection means

You give the model an image, and it finds:

- where objects are (bounding boxes)
- what those objects are (classes)

In this project, the main object detection model is:

- `data/models/Obj-Detection.onnx`

What YOLO means

YOLO is a popular object detection method:

- stands for You Only Look Once
- it scans the image once and produces many boxes + class predictions
- it is fast and good for real-time video

How YOLO and ONNX fit together here

- The model was trained or fine-tuned as a YOLO-style detector.
- Then it was exported to ONNX.
- Smart Eye loads that ONNX file and runs it with ONNX Runtime.

So:

- YOLO = the model architecture / training origin
- ONNX = the exported file format used by the app

Why ONNX is used

- ONNX is a portable model format.
- the app does not need the original training framework.
- onnxruntime can run the model on CPU or GPU.
- this makes the app simpler and more flexible.

Key files for object detection

- `backend/models/onnx_object_model.py`
- `backend/models/model_loader.py`
- `backend/pipeline/detector_manager.py`
- `backend/database/schema.sql` (`model_plugins`, `plugin_classes`, `camera_plugins`)
- `frontend/pages/models/_page.py`

5. Face Recognition in this project
-----------------------------------

What face recognition means here

- The app does not compare raw photos.
- It converts each face into a vector called an embedding, then compares embeddings.

What model is used

- It uses InsightFace.
- This is a pretrained face recognition system.
- The app loads it via `backend/models/face_model.py`.

How face recognition works

1. Detect face in the camera image.
2. Compute a face embedding vector.
3. Compare that vector to stored known-face embeddings.
4. If it matches closely enough, the face is recognized.

Important face recognition concepts

- Embedding = numeric vector representation of a face.
- Similarity = how close two embeddings are.
- Threshold = how close they must be to count as the same person.

Key files for face recognition

- `backend/models/face_model.py`
- `backend/models/model_loader.py`
- `backend/pipeline/detector_manager.py`
- `utils/embedding_utils.py`
- `backend/database/db.py` (known_faces storage)

6. The project story from scratch
---------------------------------

This section explains how the project was built from the beginning in a beginner
friendly way.

Step 1: Choose the real-world problem

The idea was to build a safety surveillance system. The project needed both:

- Computer vision AI.
- A real application that a user can operate.

The team chose safety detection because it is practical and easy to demonstrate:
the AI can detect if a person is wearing or not wearing required equipment.

Step 2: Get the dataset from Kaggle

The object detection part started with a safety dataset from Kaggle. Kaggle is a
website where people publish datasets for machine learning. For this type of
project, the dataset usually contains:

- Images of workers or people in safety environments.
- Annotation files that say where each object is in each image.
- Class names such as Hardhat, Mask, NO-Hardhat, NO-Mask, Person, and so on.

The shipped ONNX model metadata says it was trained on:

yolo_safety_dataset/data.yaml

That strongly tells us the training used YOLO format. In YOLO format, the dataset
usually has:

images/train/
images/val/
labels/train/
labels/val/
data.yaml

Each label file contains rows like:

class_id x_center y_center width height

The coordinates are normalized from 0 to 1. For example, x_center 0.5 means the
center of the object is halfway across the image width.

Step 3: Clean and prepare the dataset

Before training, the dataset needs to be checked:

- Remove broken images.
- Make sure every image has the correct matching label file.
- Make sure class IDs match the class names in data.yaml.
- Split images into training and validation sets.
- Keep the labels consistent.

The final data.yaml file would describe:

- Where the train images are.
- Where the validation images are.
- How many classes there are.
- What the class names are.

For this project, the shipped model has 10 classes:

0: Hardhat
1: Mask
2: NO-Hardhat
3: NO-Mask
4: NO-Safety Vest
5: Person
6: Safety Cone
7: Safety Vest
8: machinery
9: vehicle

Step 4: Train the object detection model

The object detector is a YOLO-style model. YOLO means "You Only Look Once".
Instead of first finding regions and then classifying them, YOLO predicts many
bounding boxes and class probabilities in one forward pass. This makes it good
for real-time camera applications.

The shipped model metadata says:

description: Ultralytics YOLO11s model trained on yolo_safety_dataset/data.yaml
author: Ultralytics
version: 8.4.14
task: detect
image size: [640, 640]
opset: 12
model input: images with shape [1, 3, 640, 640]
model output: output0 with shape [1, 14, 8400]

What that means:

- 1 means batch size 1.
- 3 means RGB color channels.
- 640 by 640 means the image is resized to 640 pixels square for the model.
- 8400 means the model checks many possible boxes.
- 14 means each prediction row includes 4 box values plus 10 class scores.

A typical Ultralytics training command would look like:

yolo task=detect mode=train model=yolo11s.pt data=yolo_safety_dataset/data.yaml epochs=100 imgsz=640 batch=16

During training, the model learns by comparing its predicted boxes with the
annotated boxes in the dataset. If it predicts the wrong box or wrong class, the
loss increases. The optimizer adjusts the neural network weights to reduce the
loss over many epochs.

Step 5: Evaluate the trained model

After training, the model should be evaluated with metrics such as:

- Precision: when the model says something exists, how often is it right?
- Recall: from all real objects, how many did the model find?
- mAP@0.5: detection quality at IoU threshold 0.5.
- mAP@0.5:0.95: stricter average across multiple IoU thresholds.
- Confusion matrix: which classes are confused with each other.

For a safety system, false negatives are important. A false negative means the
system missed a missing hardhat, missing mask, or missing vest.

Step 6: Export the trained model to ONNX

Training is usually done in PyTorch, but the app runs the model using ONNX
Runtime. ONNX is a portable model format. Exporting to ONNX makes the model easy
to load in a desktop app without needing the full training framework.

A typical export command would be:

yolo export model=runs/detect/train/weights/best.pt format=onnx opset=12 simplify=True imgsz=640

The exported file in this repository is:

data/models/Obj-Detection.onnx

Its size is about 37,941,780 bytes.

Step 7: Build an inference wrapper

After exporting the model, the app needs code to:

- Load the ONNX file.
- Choose GPU or CPU provider.
- Resize and normalize video frames.
- Run the ONNX session.
- Convert output tensors into boxes and class labels.
- Remove duplicate boxes using NMS.
- Return clean detection dictionaries to the rest of the app.

That wrapper is implemented in:

backend/models/onnx_object_model.py

Step 8: Build face recognition

Instead of training a face recognition model from scratch, the project uses
InsightFace model packs such as buffalo_l, buffalo_m, buffalo_s, buffalo_sc, or
antelopev2.

The face model does not simply store a photo and compare pixels. It converts a
face image into an embedding. An embedding is a numeric vector that represents
the identity features of the face. Similar faces should have similar vectors.

When a new known face is enrolled:

- The face model detects the face in the image.
- It calculates an embedding.
- The embedding is converted to bytes.
- The bytes are stored in the known_faces table.

When a camera sees a face:

- The face model detects the face.
- It calculates a new embedding.
- The app compares it with stored known-face embeddings.
- If the similarity is above the threshold, the app returns the person's name.

Step 9: Build the database

The project uses SQLite because it is simple, local, and works well for a desktop
application. The database file is created at runtime:

data/smarteye.db

The schema lives in:

backend/database/schema.sql

Step 10: Build the backend

The backend was built to handle:

- Cameras and playback.
- AI model loading.
- Frame processing.
- Rule evaluation.
- Alarm escalation.
- Notifications.
- Analytics.
- Database writes.

Step 11: Build the frontend

The frontend was built with PySide6. It gives the user a professional desktop
interface with sidebar navigation, dashboard, forms, tables, cards, settings,
and dialogs.

Step 12: Connect everything

The final app connects all pieces:

Camera frame -> AI models -> merged detection state -> rules -> alarms -> database -> UI.

18. Create MainWindow.
19. Show MainWindow.
20. If auto_start_cameras is enabled, start all enabled cameras shortly after launch.

11. Database overview
---------------------

Smart Eye uses a local SQLite database.

Schema file:

backend/database/schema.sql

Main database code:

backend/database/db.py

Compatibility import:

backend/repository/db.py

The repository/db.py file simply imports everything from backend/database/db.py.
This lets other code use:

from backend.repository import db

instead of importing backend.database.db directly.

The database code uses:

- SQLite WAL mode for better read/write behavior.
- A writer queue and writer thread for writes.
- One connection per thread for reads.
- Migrations to upgrade old databases.


12. Why there is a database writer thread
-----------------------------------------

SQLite can handle many reads, but concurrent writes can be tricky. Camera threads,
alarm workers, UI pages, and analytics can all touch the database.

To avoid random "database is locked" problems, backend/database/db.py queues write
operations into _write_queue. A single writer thread runs _writer_loop and performs
write operations one at a time.

That design makes the app more stable because:

- Camera processing is less likely to block.
- Alarm logging is more reliable.
- Multiple UI pages can request writes without corrupting the database.

14. Important database tables
-----------------------------

cameras
    Stores camera definitions. Important fields:
    id, name, source, enabled, face_recognition, location, resolution,
    fps_limit, face_similarity_threshold, added_at.

zones
    Stores camera zones. A zone is a rectangle area on a camera frame. The schema
    has x1, y1, x2, y2. This lets rules focus on certain areas.

model_plugins
    Stores object detection model plugins. Important fields:
    name, model_type, weight_path, enabled, confidence, preferred_provider,
    last_provider, last_error, description, version.

plugin_classes
    Stores the class list for each model plugin. For this project, examples are
    Hardhat, Mask, NO-Hardhat, NO-Mask, NO-Safety Vest, Person, Safety Cone,
    Safety Vest, machinery, vehicle.

camera_plugins
    Connects cameras to plugins. A camera can use specific plugins.

camera_plugin_classes
    Per-camera overrides for plugin classes. For example, a camera can disable
    a class or set a different confidence threshold.

rules
    Stores user-created rules. Important fields:
    name, description, logic, action, enabled, priority, camera_id, zone_id.

rule_conditions
    Stores conditions inside a rule. Example:
    attribute = NO-Hardhat
    operator = eq
    value = true

alarm_actions
    Stores what happens when a rule is triggered. Actions can include popup,
    sound, email, webhook, and log. They can also have escalation levels and
    cooldowns.

known_faces
    Stores enrolled people. It includes metadata and the embedding BLOB used for
    face recognition.

face_inbox
    Stores unknown faces captured from cameras before an admin assigns them to a
    real known face.

access_log
    Stores access decisions for faces.

detection_logs
    Stores detection history. This is one of the most important tables. It stores
    timestamp, camera_id, identity, face confidence, detections JSON, gender,
    triggered rules, alarm level, snapshot path, and reviewed flag.

notification_profiles
    Stores email and webhook targets.

app_settings
    Stores application settings as key/value rows.

clips
    Stores saved clip metadata, including path, source, camera_id, timestamp,
    face label, triggered rules, and object types.

accounts
    Stores login accounts with password hashes, salts, allowed tabs, admin flag,
    security questions, avatar path, and login timestamps.

16. Authentication and accounts
-------------------------------

Accounts are stored in the accounts table.

Passwords are not stored directly. The database code hashes passwords using:

hashlib.pbkdf2_hmac("sha256", password, salt, 120000)

That means:

- The original password is not stored.
- A random salt is stored.
- The password hash is stored.

Security answers are also hashed.

If no accounts exist, ensure_default_account creates a bootstrap admin account:

admin@smarteye.local

The password is generated and logged so the first user can log in, then they
should change it.

The frontend authentication overlay is managed in:

frontend/main_window.py
frontend/widgets/auth_overlay.py

19. CameraThread live processing
--------------------------------

CameraThread is the live camera worker. It inherits from QThread so it can run
outside the main UI thread.

Signals:

frame_ready(camera_id, frame, state)
    Sent to the UI whenever a display frame is ready.

error_occurred(camera_id, message)
    Sent when a camera fails.

reconnecting(camera_id)
    Sent when the thread tries to reconnect.

fps_updated(camera_id, fps)
    Sent every second with measured FPS.

CameraThread responsibilities:

- Open the camera source with OpenCV.
- Support numeric webcams, RTSP, HTTP streams, video-like sources, and optional
  Twitch sources through streamlink.
- Configure capture buffer and resolution.
- Read frames in a loop.
- Maintain FPS timing.
- Run inference in a separate single-worker executor.
- Avoid blocking video display while inference is running.
- Store a short frame buffer for live clip saving.
- Save clips when rules fire and clip saving is enabled.
- Emit frames to the UI with the latest detection state.
- Reconnect if a live stream temporarily fails.


20. Live camera frame flow
--------------------------

This is the main runtime flow:

1. User clicks Start All in the Dashboard, or auto_start_cameras starts cameras.
2. CameraManager creates CameraThread for each enabled camera.
3. CameraThread opens cv2.VideoCapture.
4. CameraThread reads a frame.
5. CameraThread optionally adds the frame to the live clip buffer.
6. If it is time to run inference, CameraThread submits work to its inference executor.
7. The inference worker calls DetectorManager.process_frame.
8. DetectorManager returns faces, objects, and timing.
9. inference_utils.build_state merges detections and evaluates rules.
10. PipelineService handles triggered rules, escalation, alarms, inbox capture, and heatmap updates.
11. CameraThread stores the latest state.
12. CameraThread emits frame_ready.
13. Dashboard receives frame_ready and updates the video widget, alarms, FPS, and performance panel.

24. ONNX provider selection
---------------------------

The object model can try providers such as:

- CUDAExecutionProvider
- DmlExecutionProvider
- ROCMExecutionProvider
- CoreMLExecutionProvider
- OpenVINOExecutionProvider
- CPUExecutionProvider

The code checks:

- Is GPU enabled in settings?
- Did the user choose a preferred provider?
- Which providers are available on this machine?

If GPU loading or warmup fails, it falls back to CPU. This is important because
the app should still run even when a GPU provider is unavailable.


25. Object detector preprocessing
---------------------------------

ONNXObjectModel.detect(frame) prepares frames using OpenCV:

cv2.dnn.blobFromImage(
    frame,
    scalefactor=1.0 / 255.0,
    size=(input_width, input_height),
    mean=(0, 0, 0),
    swapRB=True,
    crop=False
)

This means:

- Pixel values are divided by 255 so they become 0 to 1 floats.
- The frame is resized to the model input size.
- BGR is converted to RGB using swapRB=True.
- The shape becomes [1, 3, height, width].


26. Object detector postprocessing
----------------------------------

After ONNX Runtime returns output0, the code:

1. Converts the output to a NumPy array.
2. Handles output shape differences by transposing when needed.
3. Reads class probabilities after the first 4 box values.
4. Picks the highest class score for each row.
5. Keeps rows whose score is above the confidence threshold.
6. Converts center-based boxes into x1, y1, x2, y2 boxes.
7. Scales boxes back to the original frame size.
8. Runs cv2.dnn.NMSBoxes to remove duplicate overlapping boxes.
9. Returns detections like:

{
    "bbox": [x1, y1, x2, y2],
    "confidence": 0.91,
    "class": 2,
    "class_name": "NO-Hardhat"
}

28. Face detection and embeddings
---------------------------------

FaceModel.detect_faces(frame) does:

1. Calls InsightFace app.get(frame).
2. For each detected face, extracts bounding box.
3. Extracts the embedding vector.
4. Normalizes the embedding.
5. Optionally extracts gender and gender confidence.
6. Returns a list of face dictionaries.

FaceModel.get_embedding(frame) returns the embedding for the first detected face.
This is used during enrollment/import.


29. How face matching works
---------------------------

When the app wants to identify a face:

1. It receives the current face embedding.
2. It loads known faces from the database.
3. It converts stored embedding bytes back to NumPy arrays.
4. It stacks all known embeddings into a matrix.
5. It normalizes them.
6. It computes similarity using a dot product:

scores = known_matrix @ query_embedding

Because embeddings are normalized, the dot product behaves like cosine similarity.

7. It picks the highest score.
8. If the highest score is greater than or equal to the threshold, it returns the
   identity.

Default threshold:

face_similarity_threshold = 0.45

If a threshold is too low, the system may match the wrong person. If it is too
high, it may fail to recognize the correct person. So threshold tuning is a real
deployment step.

32. Liveness manager
--------------------

Important file:

backend/pipeline/liveness_manager.py

The liveness check is a simple anti-spoofing challenge. It is not a full deep
anti-spoofing model. It asks for face movement such as turning left then right.

The logic tracks faces per camera and tries to verify:

- Did the face turn left?
- Did the face then turn right?
- Did this happen within the challenge time?

It uses:

- Bounding box movement.
- Eye detection with OpenCV Haar cascade.
- Track state per camera.

If liveness fails, it can save a snapshot and write a detection log with:

rules_triggered = ["LivenessFailure"]
alarm_level = 2


33. DetectorManager
-------------------

Important file:

DetectorManager is the center of live AI processing.

It owns:

- The face model.
- Loaded object plugin models.
- Camera-to-plugin cache.
- Camera thresholds cache.
- Camera settings cache.

Important tasks:

34. Why frames are scaled before inference
------------------------------------------

Camera frames can be large, such as 1920x1080. Running AI on full size frames can
be slow. DetectorManager scales frames down before inference using _scale_frame.

Default maximum inference dimension:

_MAX_INFER_DIM = 512

This means the largest side is reduced to 512 pixels unless the frame is already
smaller. After detection, boxes are scaled back up to original frame coordinates.

This improves speed while keeping the UI responsive.


35. Object and face smoothing
-----------------------------

Detection boxes can jitter frame to frame. The same person may appear with boxes
that move a little even if the person is standing still.

DetectorManager uses smoothing logic:

- It remembers previous boxes.
- It calculates IoU and movement.
- It predicts and matches boxes.
- It smooths position and size changes.
- It stores velocity values track_vx and track_vy.

CameraThread can also propagate boxes briefly between inference frames so the UI
does not look frozen while waiting for the next AI result.

---------------------------

DetectorManager.process_frame returns:

{
    "faces": [...],
    "objects": [...],
    "face_time_ms": 12.3,
    "object_time_ms": 25.7
}

Then inference_utils.build_state turns that into a more complete state:

{
    "identity": "Person Name" or None,
    "gender": "male" or "female" or "unknown",
    "face_confidence": 0.92,
    "camera_id": 1,
    "face_bbox": [x1, y1, x2, y2],
    "all_faces": [...],
    "object_bboxes": [...],
    "detections": {
        "Hardhat": true,
        "NO-Hardhat": false,
        "identity": "unknown",
        "gender": "unknown"
    },
    "face_time_ms": ...,
    "object_time_ms": ...
}

This clean state is what rules use.

38. Rule engine
---------------

Important file:

backend/pipeline/rule_engine.py

Rules let non-programmers define what should count as a problem.

A rule has:

- Name.
- Description.
- Logic: AND or OR.
- Action: alarm, suppress, or log_only.
- Priority.
- Optional camera_id.
- Enabled flag.

Each rule has conditions. A condition has:

- Attribute.
- Operator.
- Value.

Supported operators include:

- eq
- neq
- gt
- lt
- gte
- lte
- contains

Examples:

NO-Hardhat eq true
NO-Mask eq true
identity eq unknown
object contains vehicle
Person gt 3

RuleEngine caches rules and conditions briefly so it does not query the database
too much during live detection.


39. AND and OR behavior
-----------------------

If a rule uses AND:

All valid conditions must be true, and unknown conditions make the rule fail
closed. This is safer because the app should not trigger an AND rule if required
information is missing.

If a rule uses OR:

Any valid true condition can trigger the rule.

Special case:

identity eq unknown is allowed to evaluate unknown identity intentionally. This
lets the user create a rule for unknown people.


40. Suppress rules
------------------

Some rules can have action = suppress.

Suppression means:

- A rule can cancel another triggered rule in certain cases.
- The engine collects suppressed rule IDs and removes them from the final
  triggered list.

This is useful for exceptions. For example, if a rule would normally alarm on
unknown identity, another rule could suppress it for a specific camera or condition.


41. Escalation manager
----------------------

Important file:

backend/pipeline/escalation_manager.py

EscalationManager tracks active violations over time per camera and rule.

It uses keys:

(camera_id, rule_id)

This prevents one camera's violation from affecting another camera.

How escalation works:

1. A rule triggers.
2. EscalationManager records when it started.
3. The violation remains active while the rule keeps triggering.
4. Alarm actions have trigger_after_sec values.
5. The active level increases when enough time has passed.

Example:

Level 1 after 0 seconds: popup
Level 2 after 10 seconds: sound
Level 3 after 30 seconds: email

This prevents every small moment from immediately becoming the highest alarm.

5. DetectorManager filters detections by confidence.
6. Analyzer creates detections["NO-Hardhat"] = true.
7. RuleEngine checks rules.
8. A rule like "NO-Hardhat eq true" passes.
9. PipelineService receives the triggered rule.
10. EscalationManager updates active violation time.
11. AlarmHandler schedules actions.

13. If configured, it writes detection_logs.
14. If snapshot_on_alarm is enabled, it saves a snapshot.

17. Logs page can show the event later.

73. How model plugins work
--------------------------

The app does not hard-code only one object model into the pipeline. It uses a
plugin-style database design.

model_plugins table stores:

- Plugin name.
- Model type.
- Weight path.
- Enabled flag.
- Confidence.
- Preferred provider.

plugin_classes table stores:

- Class index.
- Class name.
- Display name.
- Value type.
- Enabled flag.
- Confidence.
- Color.

camera_plugins table controls which cameras use which plugins.

This means the app can later support another ONNX model by registering it in the
Models page.


74. How class thresholds work
-----------------------------

There are multiple confidence levels:

- Plugin-level confidence.
- Class-level confidence.
- Camera-specific class override confidence.

DetectorManager._run_plugin decides the effective threshold. This lets a user
make a class stricter or looser for a specific camera.

Example:

On a far-away camera, hardhats may be small. The user might lower Hardhat
confidence slightly for that camera.


75. How provider fallback is stored
-----------------------------------

If a model fails on GPU and falls back to CPU, model_loader updates the plugin
row with:

- last_provider
- last_error
- last_error_at

This is useful because the Models page can show why a model is running slower.

84. Extra AI basics before reading the code
-------------------------------------------

This section is added as an extra beginner chapter. It explains common AI ideas
that appear in Smart Eye and in most computer vision projects.

Artificial intelligence, in simple words, means building software that can make
decisions that normally need human-like perception or judgment.

In Smart Eye, AI is used for:

1. Looking at an image.
2. Finding objects inside it.
3. Finding faces inside it.
4. Comparing faces to known people.
5. Giving confidence scores so the app can decide what to do.

The important thing to understand is this:

AI models do not magically know what a hardhat, mask, vest, or face is. They need
training or pretraining.

For object detection, our model was trained using labeled examples. A labeled
example is an image plus information about what is inside the image.

Example:

Image:
    worker_001.jpg

Labels:
    Person at box [120, 40, 450, 600]
    Hardhat at box [180, 30, 310, 120]
    Safety Vest at box [150, 240, 380, 520]

The model sees many examples like this. At first it predicts badly. During
training, it compares its predictions to the correct labels. The training
algorithm adjusts the model weights so the next predictions become better.


85. What is an AI model?
------------------------

An AI model is a mathematical function with many internal numbers called weights.

You can imagine it like this:

input image -> model -> prediction

For Smart Eye object detection:

camera frame -> YOLO model -> boxes, classes, confidence scores

For Smart Eye face recognition:

face image -> InsightFace model -> face embedding vector

The important parts are:

Input:
    The data the model receives, such as an image.

Weights:
    Internal learned numbers. These are the "knowledge" of the model.

Output:
    The prediction, such as bounding boxes or an embedding.

Training:
    The process that creates or improves the weights.

Inference:
    Using an already trained model to make predictions.

The Smart Eye app mainly performs inference. It loads trained model files and
runs them on camera frames.


86. Training vs inference
-------------------------

Training and inference are different.

Training:

- Needs a dataset.
- Needs labels.
- Usually needs a powerful GPU.
- Takes time.
- Changes model weights.
- Produces a trained model file.

Inference:

- Uses an already trained model.
- Takes one new image or frame.
- Produces predictions.
- Does not change model weights.
- Must be fast for live cameras.

In the Smart Eye workflow:

1. Dataset was collected from Kaggle.
2. YOLO model was trained outside the final app.
3. Trained model was exported to ONNX.
4. ONNX model was placed in data/models.
5. The app loads the ONNX model and runs inference.

So if someone asks, "Does the app train the model every time it opens?", the
answer is no. The app only loads and runs the already trained model.


87. Common AI training terms
----------------------------

Dataset
    The collection of examples used for training and testing.

Label
    The correct answer for an example. In object detection, labels include class
    names and bounding boxes.

Class
    A category the model can predict, such as Hardhat or NO-Mask.

Training set
    Images used to teach the model.

Validation set
    Images used to check if the model is learning well.

Test set
    Images used for final evaluation after training decisions are done.

Epoch
    One full pass through the training dataset.

Batch
    A group of images processed together during training.

Loss
    A number that tells how wrong the model is. Lower loss is usually better.

Optimizer
    The algorithm that updates the model weights to reduce loss.

Learning rate
    Controls how big each training update is.

Overfitting
    When the model memorizes training images but performs badly on new images.

Augmentation
    Random changes to training images, such as flipping, scaling, brightness
    changes, or cropping. This helps the model generalize.

Precision
    Of all detections the model predicted, how many were correct?

Recall
    Of all real objects, how many did the model find?

mAP
    Mean Average Precision. A standard score for object detection quality.

Confidence
    How sure the model is about a prediction.

IoU
    Intersection over Union. It measures how much two boxes overlap.

NMS
    Non-Maximum Suppression. It removes duplicate boxes around the same object.


88. What the YOLO object model learns
-------------------------------------

The object model does not store a list of pictures. It learns visual patterns.

For example:

Hardhat:
    Rounded helmet shape on top of a person's head, often yellow/white/orange.

Mask:
    Object covering lower part of face.

Safety Vest:
    Bright vest shape on torso.

NO-Hardhat:
    Person/head area where required hardhat is missing.

NO-Mask:
    Face visible without mask.

Vehicle:
    Shape features of cars, trucks, or industrial vehicles.

The model converts pixels into features. Early layers may detect edges and simple
textures. Deeper layers combine those into object parts and full objects.

When it sees a frame, YOLO predicts many possible boxes. Each box has:

- x center
- y center
- width
- height
- class scores

The app later converts those predictions into final boxes.


89. Example training files for a YOLO dataset
---------------------------------------------

The training dataset usually has a file called data.yaml.

Example:

train: C:/datasets/yolo_safety_dataset/images/train
val: C:/datasets/yolo_safety_dataset/images/val

nc: 10
names:
  0: Hardhat
  1: Mask
  2: NO-Hardhat
  3: NO-Mask
  4: NO-Safety Vest
  5: Person
  6: Safety Cone
  7: Safety Vest
  8: machinery
  9: vehicle

A YOLO label file might look like this:

5 0.512 0.544 0.260 0.700
0 0.510 0.170 0.120 0.090
7 0.510 0.520 0.210 0.340

This means:

Line 1:
    Class 5, Person, with normalized box values.

Line 2:
    Class 0, Hardhat.

Line 3:
    Class 7, Safety Vest.

The numbers are normalized, so they are relative to the image width and height.


90. Example training commands
-----------------------------

This is a typical training command for Ultralytics:

yolo task=detect mode=train model=yolo11s.pt data=yolo_safety_dataset/data.yaml epochs=100 imgsz=640 batch=16

Meaning:

task=detect
    Train an object detection model.

mode=train
    Run training mode.

model=yolo11s.pt
    Start from a pretrained YOLO11 small model.

data=...
    Use our dataset configuration.

epochs=100
    Go through the training dataset 100 times.

imgsz=640
    Resize images to 640 by 640.

batch=16
    Train on 16 images at a time.

After training, Ultralytics normally creates:

runs/detect/train/weights/best.pt
runs/detect/train/weights/last.pt

best.pt is usually the model checkpoint with the best validation score.


91. Example export command
--------------------------

The app uses ONNX, so the trained PyTorch model is exported:

yolo export model=runs/detect/train/weights/best.pt format=onnx opset=12 simplify=True imgsz=640

This creates an ONNX file.

In this repository, the final file is:

data/models/Obj-Detection.onnx

Why export?

- ONNX Runtime is fast.
- ONNX is portable.
- The app does not need the full training code.
- The same model can run with CPU or GPU providers.


92. Why the model output is [1, 14, 8400]
-----------------------------------------

The shipped object model output is:

[1, 14, 8400]

Breakdown:

1
    One image at a time.

14
    4 box numbers plus 10 class scores.

8400
    Many candidate predictions from the YOLO detection heads.

The code transposes this into a shape that is easier to process. Then it reads:

- First 4 values as box geometry.
- Remaining 10 values as class probabilities.


93. Simplified ONNX model loading code
--------------------------------------

The real code is in:

backend/models/onnx_object_model.py

Here is a simplified version of the loading idea:

class ONNXObjectModel:
    def __init__(self, weight_path, confidence=0.6):
        self._weight_path = weight_path
        self._confidence = confidence
        self._session = None
        self._input_name = None
        self._input_shape = None
        self._class_names = {}

    def load(self):
        import onnxruntime as ort

        if not os.path.isfile(self._weight_path):
            raise MissingModelFile(self._weight_path)

        providers = ["DmlExecutionProvider", "CPUExecutionProvider"]
        self._session = ort.InferenceSession(
            self._weight_path,
            providers=providers
        )

        input_info = self._session.get_inputs()[0]
        self._input_name = input_info.name
        self._input_shape = (640, 640)

What this means:

- The class stores the model path.
- load() checks that the file exists.
- It creates an ONNX Runtime session.
- It reads the input name.
- It stores the input size.

The real project code is more advanced because it:

- Reads available providers.
- Honors GPU settings.
- Falls back to CPU when GPU fails.
- Warms the model up.
- Reads class names from ONNX metadata.
- Stores provider errors in the database.


94. Real provider selection idea

95. Simplified object detection code
------------------------------------

The real detect function is in:

Simplified:

def detect(self, frame):
    h, w = frame.shape[:2]
    inp_w, inp_h = self._input_shape

    blob = cv2.dnn.blobFromImage(
        frame,
        scalefactor=1.0 / 255.0,
        size=(inp_w, inp_h),
        swapRB=True,
        crop=False
    )

    outputs = self._session.run(None, {self._input_name: blob})
    output = outputs[0]

    detections = postprocess(output, original_width=w, original_height=h)
    return detections

Line by line:

    Get original frame height and width.

    Get the size the model expects.

cv2.dnn.blobFromImage(...)
    Resize the frame, normalize pixels, convert BGR to RGB, and put channels in
    the format the model expects.

self._session.run(...)
    Run the ONNX model.

postprocess(...)
    Convert raw model output to readable boxes and labels.


96. Simplified postprocessing code
----------------------------------

After the model returns raw output, the project does something like this:

class_probs = out[:, 4:]
class_ids = np.argmax(class_probs, axis=1)
class_scores = class_probs[np.arange(class_probs.shape[0]), class_ids]
keep = class_scores >= self._confidence

This means:

class_probs = out[:, 4:]
    Ignore the first 4 box values and keep only class scores.

np.argmax(...)
    Pick the class with the highest score for each predicted box.

class_scores
    Get the score for the chosen class.

keep
    Keep only predictions above the confidence threshold.

Then boxes are converted:

cx = kept[:, 0] * scale_x
cy = kept[:, 1] * scale_y
bw = kept[:, 2] * scale_x
bh = kept[:, 3] * scale_y

x1 = cx - bw / 2
y1 = cy - bh / 2
x2 = cx + bw / 2
y2 = cy + bh / 2

YOLO predicts center x, center y, width, height. The UI needs corner coordinates,
so the app converts them.

Then NMS removes duplicates:

idxs = cv2.dnn.NMSBoxes(boxes_xywh, scores, confidence, 0.45)

NMS example:

If the model predicts three overlapping boxes around the same hardhat, NMS keeps
the strongest one and removes the weaker duplicates.

99. Face identification code idea
---------------------------------

Simplified from FaceModel.identify:

def identify(self, embedding, threshold=0.45):
    if self._known_cache_dirty:
        self._rebuild_known_matrix()

    query = normalize(embedding)
    scores = self._known_matrix @ query

    best_index = int(np.argmax(scores))
    best_score = float(scores[best_index])

    if best_score >= threshold:
        row = self._known_rows[best_index]
        return {"id": row["id"], "name": row["name"]}, best_score

    return None, best_score

Explanation:

self._known_matrix
    A matrix containing all known face embeddings.

query
    The current face embedding.

self._known_matrix @ query
    Calculates similarity between current face and every known face very quickly.

np.argmax(scores)
    Finds the best match.

threshold
    Minimum similarity required to accept the match.

If best_score is too low, the face is considered unknown.


100. Why embeddings are better than comparing photos
----------------------------------------------------

Comparing raw photos is weak because:

- Lighting changes.
- Face angle changes.
- Distance changes.
- Camera quality changes.
- Background changes.
- Image size changes.

Embeddings solve this by converting a face to a compact numeric identity
representation.

Example:

Photo A of Nour -> [0.12, -0.08, 0.44, ...]
Photo B of Nour -> [0.11, -0.07, 0.42, ...]
Photo of someone else -> [-0.30, 0.51, 0.02, ...]

The two embeddings of the same person should be close. Different people should
be farther apart.

def cosine_similarity(a, b):
    a = a / np.linalg.norm(a)
    b = b / np.linalg.norm(b)
    return float(np.dot(a, b))

102. Database writer code explained
-----------------------------------

The real code is in:

backend/database/db.py

Simplified:

_write_queue = queue.Queue()

def _writer_loop():
    conn = _create_conn()

    while True:
        item = _write_queue.get()
        if item is None:
            break

        sql, params, result_q = item

        try:
            cur = conn.execute(sql, params)
            conn.commit()
            result_q.put((cur, None))
        except Exception as e:
            conn.rollback()
            result_q.put((None, e))

def _write_execute(sql, params=()):
    result_q = queue.Queue(maxsize=1)
    _write_queue.put((sql, params, result_q))
    cur, err = result_q.get()

    if err:
        raise err

    return cur

Explanation:

- Many parts of the app may want to write to SQLite.
- Instead of letting all threads write directly, writes go into a queue.
- One writer thread performs writes one by one.
- This reduces database locked errors.

This is a strong design choice for a camera app because camera, alarm, and UI
threads can all be active at the same time.


103. Account password hashing code explained
--------------------------------------------

Simplified from backend/database/db.py:

def _hash_password(password, salt=None):
    salt_val = salt or secrets.token_hex(16)
    digest = hashlib.pbkdf2_hmac(
        "sha256",
        password.encode("utf-8"),
        bytes.fromhex(salt_val),
        120000
    )
    return salt_val, digest.hex()

Meaning:

salt_val
    Random value added to the password before hashing.

pbkdf2_hmac
    A slow password hashing algorithm. Slow is good here because it makes brute
    force attacks harder.

120000
    Number of hashing iterations.

digest.hex()
    Store the final hash as text.

When a user logs in, the app hashes the entered password with the stored salt and
checks if the hash matches.

def merge_results(detection_results, camera_id):
    state = {
        "identity": None,
        "camera_id": camera_id,
        "all_faces": [],
        "object_bboxes": [],
        "detections": {}
    }

    faces = detection_results.get("faces", [])
    if faces:
        best_face = max(faces, key=lambda f: f.get("confidence", 0))
        if best_face.get("identity"):
            state["identity"] = best_face["identity"]["name"]

    objects = detection_results.get("objects", [])
    for obj in objects:
        class_name = obj["class_name"]
        state["detections"][class_name] = True
        state["object_bboxes"].append(obj)

    if state["identity"]:
        state["detections"]["identity"] = state["identity"]
    else:
        state["detections"]["identity"] = "unknown"

    return state

Why this matters:

The AI model output is too raw for rules. Rules need simple facts:

- Is NO-Hardhat present?
- Is NO-Mask present?
- Is identity unknown?
- How many persons are visible?

Analyzer creates those facts.


110. Rule condition code explained
----------------------------------

The real code is in:

backend/pipeline/rule_engine.py

Simplified:

def _evaluate_condition(actual, operator, expected):
    if operator == "eq":
        return actual == expected

    if operator == "neq":
        return actual != expected

    if operator == "gt":
        return actual > expected

    if operator == "contains":
        return expected in actual

    return False

Example:

actual = True
operator = "eq"
expected = True

Result:

True

Another example:

actual = "unknown"
operator = "eq"
expected = "unknown"

Result:

True

This is how a rule like "identity equals unknown" can trigger.


111. RuleEngine.evaluate_rules simplified
-----------------------------------------

Simplified:

def evaluate_rules(self, state, camera_id=None):
    rules = db.get_rules(enabled_only=True, camera_id=camera_id)
    triggered = []

    for rule in rules:
        conditions = db.get_rule_conditions(rule["id"])
        results = []

        for cond in conditions:
            actual = state["detections"].get(cond["attribute"])
            passed = _evaluate_condition(
                actual,
                cond["operator"],
                cond["value"]
            )
            results.append(passed)

        if rule["logic"] == "AND":
            passed_rule = all(results)
        else:
            passed_rule = any(results)

        if passed_rule:
            triggered.append(rule)

    return triggered

The real code is more careful because it:

- Handles object conditions.
- Handles numbers and booleans.
- Handles unknown values.
- Handles suppress rules.
- Caches rules for performance.
- Sorts by priority.

119. Common AI misunderstandings
--------------------------------

Misunderstanding:
    "The AI is always correct."

Reality:
    AI gives predictions with confidence scores. It can be wrong.

Misunderstanding:
    "If the model works on one video, it works everywhere."

Reality:
    New lighting, camera angle, uniforms, distance, and motion blur can reduce
    accuracy.

Misunderstanding:
    "Higher confidence threshold is always better."

Reality:
    Higher threshold reduces false positives but may increase missed detections.

Misunderstanding:
    "Lower confidence threshold is always better."

Reality:
    Lower threshold catches more objects but may create false alarms.

Misunderstanding:
    "Face recognition means storing photos."

Reality:
    The important recognition data is the face embedding, not the raw photo.

Misunderstanding:
    "The app trains while cameras run."

Reality:
    The app performs inference. Training happened before the ONNX model was added.


120. How to explain ONNX to a beginner
--------------------------------------

ONNX is like a saved version of a trained model in a common format.

Analogy:

Training framework:
    Like the workshop where the model is built.

ONNX file:
    Like the finished machine exported from the workshop.

ONNX Runtime:
    Like the engine that runs the finished machine.

Smart Eye does not need the whole training workshop to run the model. It only
needs the exported ONNX file and ONNX Runtime.


121. How to explain confidence thresholds
-----------------------------------------

Imagine the model says:

NO-Hardhat confidence = 0.92

If threshold is 0.60:

0.92 >= 0.60, so keep it.

If threshold is 0.95:

0.92 < 0.95, so ignore it.

Threshold controls sensitivity.

In Smart Eye, thresholds can exist at multiple levels:

- Plugin default confidence.
- Class confidence.
- Camera class override confidence.
- Face similarity threshold.


122. How to explain NMS visually
--------------------------------

Suppose the model predicts:

Box A around hardhat, confidence 0.91
Box B around same hardhat, confidence 0.83
Box C around same hardhat, confidence 0.77

Without NMS:
    The UI might draw three boxes around one object.

With NMS:
    The app keeps Box A and removes B and C.

This makes detections cleaner.


123. How to explain face threshold
----------------------------------

Suppose a detected face is compared to known people:

Ahmed score = 0.72
Mona score = 0.31
Omar score = 0.22

If threshold is 0.45:

Best score is 0.72, so identity is Ahmed.

If threshold is 0.80:

Best score is 0.72, but it is below 0.80, so identity is unknown.

This is why threshold tuning matters.

126. Why the project separates backend and frontend
---------------------------------------------------

Backend:
    Does the work.

Frontend:
    Shows the work and lets users control it.

Why separate them?

- Easier to understand.
- Easier to test.
- UI changes do not break AI code as easily.
- AI code can run in threads without mixing with UI drawing.
- Database logic stays in one place.

Example:

The Dashboard does not directly run the ONNX model. It receives already processed
state from CameraThread. This keeps the UI cleaner.


127. Why the project uses JSON inside some database fields
----------------------------------------------------------

Some data is flexible. For example, detections can contain many different class
names depending on the model plugin.

Instead of creating a new database column for every possible class, the app stores
some data as JSON text.

Example detection JSON:

{
    "NO-Hardhat": true,
    "NO-Mask": false,
    "Person": 2,
    "identity": "unknown",
    "gender": "unknown"
}

Benefits:

- Flexible.
- Easy to store changing detection facts.
- Works with different models.

Tradeoff:

- Harder to query than normal columns.

That is why the app also stores important common fields separately, such as:

- identity
- face_confidence
- gender_norm
- alarm_level
- has_identity


128. Why logs and snapshots are both needed
-------------------------------------------

Log row:
    Structured data for searching, filtering, reports, and analytics.

Snapshot:
    Visual proof of what happened.

Example:

The log says:
    Camera 2, NO-Hardhat, alarm level 2, 10:30 AM.

The snapshot shows:
    The actual frame where the missing hardhat was detected.

Both are useful.


129. Why the app has settings in the database
---------------------------------------------

Settings are stored in app_settings instead of hard-coded because users may need
to change behavior without editing code.

Examples:

- Change theme.
- Enable GPU.
- Change face threshold.
- Enable/disable liveness.
- Change SMTP host.
- Limit CPU/RAM.
- Enable auto-start cameras.
- Change log retention.

The config helper reads settings and caches them for performance.

131. How a safety rule is added from code
-----------------------------------------

Example:

rule_id = db.add_rule(
    name="Missing Hardhat",
    description="Trigger when a hardhat violation is detected",
    logic="AND",
    action="alarm",
    priority=10,
    camera_id=None
)

db.add_rule_condition(
    rule_id=rule_id,
    attribute="NO-Hardhat",
    operator="eq",
    value="true"
)

db.add_alarm_action(
    rule_id=rule_id,
    escalation_level=1,
    trigger_after_sec=0,
    action_type="popup",
    action_value="",
    cooldown_sec=10
)

db.add_alarm_action(
    rule_id=rule_id,
    escalation_level=2,
    trigger_after_sec=5,
    action_type="sound",
    action_value="",
    cooldown_sec=10
)

Meaning:

- If NO-Hardhat becomes true, the rule triggers.
- Immediately show popup.
- If it continues for 5 seconds, play sound.

133. Why code uses try/except a lot
-----------------------------------

Camera and AI apps deal with many things that can fail:

- Camera disconnected.
- Model file missing.
- GPU provider unavailable.
- Database locked or unavailable.
- Bad image.
- Bad video file.
- Network webhook failed.
- SMTP credentials wrong.
- User deleted a file.

The app catches exceptions so one failure does not crash the entire program.

Example:

try:
    self._session = ort.InferenceSession(model_path, providers=providers)
except Exception:
    self._session = ort.InferenceSession(model_path, providers=["CPUExecutionProvider"])

Meaning:

Try the preferred way first. If it fails, use a safer fallback.


134. Why the app uses caches
----------------------------

The app reads settings, rules, plugin classes, and known faces often.

Reading from the database every frame would be expensive. So the app caches data
for a short time.

Examples:

- RuleEngine caches rules for about 1 second.
- Analyzer caches plugin classes.
- FaceModel caches known face embeddings until changed.
- DetectorManager caches camera thresholds and settings.
- config.py caches app settings.

Cache means:

Remember the value for a little while, then reuse it.

When data changes, the app invalidates caches.


135. Why the app uses cooldowns
-------------------------------

Without cooldowns, one violation could send hundreds of emails or logs.

Example:

Camera runs at 30 FPS.
Missing hardhat lasts 10 seconds.
That is 300 frames.

If the app sent one email per frame, it would send 300 emails.

Cooldown prevents this.

Alarm actions have cooldown_sec, such as 10 seconds. This means after an action
runs, it cannot run again until cooldown time passes.


136. What happens when a model is wrong
---------------------------------------

AI can make mistakes:

False positive:
    Model says NO-Hardhat, but the person actually has a hardhat.

False negative:
    Model misses a real NO-Hardhat violation.

Why mistakes happen:

- Poor lighting.
- Low camera resolution.
- Motion blur.
- Object is too small.
- Unusual helmet or vest design.
- Person is partially hidden.
- Dataset did not include enough similar examples.

How to improve:

- Add more training data.
- Improve labels.
- Retrain model.
- Tune confidence thresholds.
- Use better camera placement.
- Use higher quality video.
- Add validation tests.


137. How to explain the AI part in one minute
---------------------------------------------

"The AI part has two main models. First, an object detection model based on
YOLO11s was trained on a Kaggle safety dataset. It learned classes like Hardhat,
Mask, NO-Hardhat, NO-Mask, Safety Vest, Person, machinery, and vehicle. After
training, we exported it to ONNX, and the app runs it with ONNX Runtime.

Second, face recognition uses InsightFace. It detects faces and converts each
face into an embedding vector. We store known people embeddings in SQLite. When a
camera sees a face, we compare the new embedding to stored embeddings. If the
similarity is above the threshold, we identify the person; otherwise the person
is unknown."


138. How to explain the code part in one minute
-----------------------------------------------

"main.py starts the app, initializes folders, logging, database, settings, and the
main PySide6 window. The backend handles cameras, models, rules, alarms, and
database operations. Each camera runs in a CameraThread. Frames go to
DetectorManager, which runs the object model and face model. Analyzer creates a
clean state. RuleEngine checks the state against user rules. PipelineService and
AlarmHandler handle actions. The frontend pages display live feeds, cameras,
models, faces, rules, logs, playback, analytics, notifications, and settings."

140. Final expanded mental model
--------------------------------

The project can be remembered as three connected stories:

Story 1: AI model story
    Dataset from Kaggle, labels, training, validation, export to ONNX, inference
    in the app.

Story 2: Backend story
    Cameras, frames, model inference, face recognition, analyzer, rules, alarms,

    PySide6 pages let the user control everything without writing code.

Together:

Dataset teaches the model.
Model watches the camera.
